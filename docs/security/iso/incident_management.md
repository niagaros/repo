# Incident Management Checklist

---

## 1. Purpose
This checklist defines the required steps for handling incidents efficiently and consistentlyâ€”reducing risk, minimizing impact, and restoring normal operations.

---

## 2. Scope
This applies to all incidents affecting:

- Systems, data, or infrastructure
- Customers or internal users
- Security, quality, or operational continuity

---

## 3. Incident Definition
An **incident** is any unplanned interruption, degradation of service, or security event that affects normal operation.

---

## Incident Management Checklist

### A. Detection & Reporting

| Step | Task | Owner | Status |
|------|-------|--------|--------|
| 1 | Identify incident source (monitoring alert, user report, tool) |  |
| 2 | Log initial incident in tracking system |  |
| 3 | Assign incident ID and severity level (Low/Med/High/Critical) |  |
| 4 | Notify responsible parties (team lead / on-call / management) |  |

---

### B. Assessment & Categorization

| Step | Task | Owner | Status |
|------|-------|--------|--------|
| 5 | Validate incident |  |
| 6 | Determine impact (users, systems, data) |  |
| 7 | Determine urgency (time sensitivity) |  |
| 8 | Categorize as Incident / Major Incident / Security Event |  |

---

### C. Containment & Resolution

| Step | Task | Owner | Status |
|------|-------|--------|--------|
| 9 | Implement temporary containment (stop the bleeding) |  |
|10 | Root cause investigation begins |  |
|11 | Apply fix or workaround |  |
|12 | Validate service restoration |  |

---

### D. Communication

| Step | Task | Owner | Status |
|------|-------|--------|--------|
|13 | Update stakeholders (internal/external) |  |
|14 | Document timestamps and progress updates |  |
|15 | If critical: send status updates at agreed intervals |  |

---

### E. Closure

| Step | Task | Owner | Status |
|------|-------|--------|--------|
|16 | Confirm normal operations restored |  |
|17 | Document incident summary and timeline |  |
|18 | Record lessons learned |  |
|19 | Close incident ticket |  |

---

## Post-Incident Review (PIR)

| Requirement | Description |
|-------------|-------------|
| **Root cause identified** | What caused the incident? Why did it occur? |
| **Corrective actions defined** | What must change to prevent recurrence? |
| **Preventive actions defined** | What can be improved proactively? |
| **Evidence attached** | Logs, screenshots, reports |

---

## 4. Roles

| Role | Responsibility |
|------|----------------|
| Incident Manager | Oversees incident handling and communication |
| Technical Owner | Performs investigation and resolution |
| Communications Lead | Sends external/internal updates |
| Post-Incident Lead | Conducts PIR + root cause documentation |

---

## 5. Evidence & Records

Artifacts that must be retained:

- Incident ticket and timeline
- Root cause analysis (RCA)
- Action plan
- Closure approval
